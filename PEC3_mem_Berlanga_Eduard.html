1. INTRODUCCIÓN

1.1. Contexto y justificación

Tras completar el desarrollo del backend en la PEC 2, el proyecto RunTogether requiere una interfaz de usuario que permita a los corredores y organizadores interactuar con el sistema de forma intuitiva. Esta fase (PEC 3) se centra en "dar cara" a la lógica de negocio, creando una experiencia web moderna y accesible.

En el panorama actual de aplicaciones deportivas, la experiencia de usuario (UX) es un factor diferenciador clave. Los usuarios esperan interfaces rápidas, reactivas y que funcionen perfectamente en dispositivos móviles. Por ello, se ha optado por una arquitectura SPA (Single Page Application) que elimina las recargas de página y ofrece una sensación de "app nativa".

1.1.1. Análisis de Referentes

Para definir la línea gráfica y funcional, se han analizado plataformas líderes en el sector:

  • Strava: Destaca por su uso de mapas interactivos y feed de actividad social. Se toma como referencia su limpieza visual y el uso de espacios en blanco.
  • Meetup: Referente en la organización de eventos grupales. Se adopta su claridad en la presentación de fechas, horarios y ubicaciones en las tarjetas de evento.

1.2. Objetivos de la PEC 3

Objetivo General: Desarrollar una aplicación web SPA (Single Page Application) con React que consuma la API de RunTogether y proporcione una experiencia de usuario fluida y responsive.

Objetivos Específicos

  • Configurar el entorno frontend con Vite y React, asegurando un build optimizado.
  • Implementar la navegación y rutas protegidas (React Router) para gestionar el acceso según roles.
  • Desarrollar componentes reutilizables (tarjetas, formularios, botones) siguiendo el principio DRY (Don't Repeat Yourself).
  • Integrar los servicios de la API (Axios) para autenticación y datos, manejando estados de carga y error.
  • Aplicar un diseño visual coherente y atractivo mediante TailwindCSS.

1.3. Impacto en la experiencia de usuario

Una buena interfaz reduce la barrera de entrada para nuevos usuarios. Se ha priorizado la claridad visual, la retroalimentación inmediata (loading states, mensajes de éxito/error) y la adaptabilidad a dispositivos móviles, dado que muchos usuarios accederán desde sus teléfonos antes de una carrera.

Justificación de Tecnologías

  • Vite vs CRA: Se eligió Vite por su velocidad de arranque y HMR (Hot Module Replacement) instantáneo. Utiliza ES Modules nativos en desarrollo, lo que elimina los tiempos de espera de empaquetado.
  • TailwindCSS: Permite un desarrollo rápido sin salir del HTML (JSX). Su enfoque "utility-first" facilita la creación de diseños responsive sin escribir media queries manuales en archivos CSS separados.
  • Context API: Para una aplicación de este tamaño, Redux o Zustand hubieran añadido complejidad innecesaria. Context API combinado con el hook useReducer es suficiente para gestionar el estado global de autenticación y preferencias.

Estrategia de Control de Versiones

Se ha seguido un flujo de trabajo basado en Gitflow simplificado:

  • main: Rama de producción, código estable.
  • develop: Rama de integración de nuevas características.
  • feature/*: Ramas temporales para cada nueva funcionalidad (ej: feature/login-page).

2.3. Gestión de Estado y Rutas

El estado de autenticación es crítico. Se implementó un AuthProvider que envuelve la aplicación y persiste la sesión mediante localStorage.

// src/context/AuthContext.jsx export const AuthProvider = ({ children }) => { const [user, setUser] = useState(null); const [loading, setLoading] = useState(true); useEffect(() => { // Verificar token al cargar la app const token = localStorage.getItem('token'); if (token) { api.get('/auth/me') .then(res => setUser(res.data)) .catch(() => localStorage.removeItem('token')) .finally(() => setLoading(false)); } else { setLoading(false); } }, []); const login = async (credentials) => { const { data } = await api.post('/auth/login', credentials); localStorage.setItem('token', data.token); setUser(data.user); }; return ( {!loading && children} ); };

Hooks Personalizados

Para abstraer la lógica de negocio de la vista, se crearon Custom Hooks. Por ejemplo, useEvents encapsula la lógica de obtención y filtrado de eventos.

3.2. Integración con API

La comunicación con el backend se centraliza en una instancia de Axios configurada con interceptores para manejar la seguridad de forma transparente:

// src/services/api.js const api = axios.create({ baseURL: import.meta.env.VITE_API_URL }); // Interceptor de Request: Adjuntar Token api.interceptors.request.use(config => { const token = localStorage.getItem('token'); if (token) { config.headers.Authorization = `Bearer ${token}`; } return config; }); // Interceptor de Response: Manejo de Errores Globales api.interceptors.response.use( response => response, error => { if (error.response?.status === 401) { localStorage.removeItem('token'); window.location.href = '/login'; } return Promise.reject(error); } );

3.3. Accesibilidad y Buenas Prácticas

Se ha prestado especial atención a la accesibilidad (a11y) para asegurar que la plataforma sea utilizable por el mayor número de personas:

  • HTML Semántico: Uso correcto de etiquetas <main>, <nav>, <article>.
  • Contraste: Verificación de colores para cumplir con el ratio AA de las WCAG.
  • Navegación por teclado: Todos los elementos interactivos tienen estados :focus visibles.
  • Atributos ARIA: Uso de aria-label en botones que solo contienen iconos.

3.4. Pruebas de Interfaz

Se han realizado pruebas manuales exhaustivas para validar los flujos de usuario:

Matriz de Pruebas Frontend

  • Autenticación:
    • Login correcto redirige a Dashboard.
    • Login fallido muestra mensaje de error (Toast).
    • Persistencia de sesión al recargar página (F5).
  • Navegación:
    • Intento de acceso a /create-event sin login redirige a /login.
    • 404 Not Found implementado para rutas inexistentes.
  • Formularios:
    • Validación en tiempo real (email inválido, campos vacíos).
    • Deshabilitado de botón "Enviar" durante la petición (loading state).

4. CONCLUSIONES Y TRABAJOS FUTUROS

4.1. Retos Encontrados

Durante el desarrollo surgieron varios desafíos técnicos que fueron resueltos:

  • Gestión de Tokens: La expiración del token JWT requería una estrategia para limpiar el estado local y redirigir al usuario. Se solucionó mediante interceptores de Axios.
  • Renderizado Condicional: Mostrar diferentes opciones de menú según el rol del usuario (Admin vs Runner) implicó crear lógica condicional en el componente Navbar.
  • Carga de Imágenes: Optimizar la carga de imágenes de eventos para no bloquear el renderizado inicial.

4.2. Conclusiones Generales

La PEC 3 ha sido completada con éxito. La aplicación React es funcional, rápida y estéticamente agradable. La separación de responsabilidades entre Backend y Frontend ha permitido trabajar de forma modular y escalable.

El uso de Vite ha mejorado significativamente la experiencia de desarrollo (HMR instantáneo) comparado con CRA, permitiendo iteraciones más rápidas.

4.3. Trabajo Futuro (PEC 4)

  • Tests E2E: Implementar Cypress para probar flujos completos automáticamente.
  • Despliegue: Publicar el frontend en Vercel o Netlify y el backend en Render/Railway.
  • Optimización: Lazy loading de imágenes y componentes para mejorar el rendimiento (Code Splitting).
  • PWA: Convertir la web en una Progressive Web App para permitir instalación en móviles y funcionamiento offline básico.

5. GLOSARIO

SPA (Single Page Application)
Aplicación web que carga una sola página HTML y actualiza dinámicamente el contenido según la interacción del usuario.

Componente
Pieza de código independiente y reutilizable que representa una parte de la interfaz de usuario.

Hook
Función especial de React que permite "engancharse" a características como el estado y el ciclo de vida.

JWT (JSON Web Token)
Estándar abierto para la creación de tokens de acceso que permiten la propagación de identidad y privilegios.
