1. INTRODUCCIÓN

1.1. Contexto y justificación del Trabajo

RunTogether es una plataforma web diseñada para conectar organizadores de eventos de running con corredores de distintos niveles. En esta PEC 3 el foco se sitúa en el desarrollo del frontend en React, construyendo una interfaz moderna, accesible y fácil de usar que consuma la API desarrollada en la PEC 2.

1.2. Objetivos del Trabajo

El objetivo principal es implementar una interfaz web que permita a los usuarios:

  • Consultar y filtrar eventos de running de forma intuitiva.
  • Registrarse e iniciar sesión en la plataforma.
  • Inscribirse en eventos y revisar sus inscripciones.
  • Disfrutar de una experiencia consistente en escritorio y móvil.

1.3. Impacto y enfoque

El desarrollo del frontend se ha realizado priorizando la usabilidad, la accesibilidad y la separación clara entre presentación y lógica de negocio. Se emplean componentes reutilizables y un diseño coherente con la identidad visual definida para RunTogether.

React 19

Biblioteca UI

Vite

Build Tool

Tailwind CSS

Framework CSS

Axios

Cliente HTTP

2.2. Arquitectura del Frontend

La aplicación es una Single Page Application (SPA) construida con React. La arquitectura sigue una estructura modular basada en características (features) y componentes reutilizables.

  • Componentes: Bloques de UI reutilizables (Botones, Tarjetas, Inputs).
  • Páginas: Vistas principales que componen la estructura de navegación.
  • Contexto: Estado global para autenticación y preferencias de usuario.
  • Servicios: Capa de abstracción para la comunicación con la API.

2.3. Estructura de Componentes

frontend/src/ ├── components/ # UI reutilizable (Button, Card, Navbar) ├── pages/ # Vistas (Home, Login, EventDetails) ├── context/ # AuthContext, ThemeContext ├── services/ # Llamadas a API (authService, eventService) ├── hooks/ # Custom Hooks (useAuth, useFetch) ├── assets/ # Imágenes y estilos globales └── App.jsx # Configuración de Rutas

2.4. Sistema de Rutas y Navegación

Se utiliza React Router para gestionar la navegación sin recargar la página. Se han definido:

  • Rutas Públicas: Home, Login, Registro, Listado de Eventos.
  • Rutas Protegidas: Crear Evento, Mi Perfil, Mis Inscripciones (requieren Token JWT).

2.5. Gestión de Estado y Consumo de la API

La seguridad y el estado del usuario se gestionan mediante Context API. Al iniciar sesión, el token JWT recibido del backend se almacena en localStorage y se utiliza para configurar los headers de autorización en Axios.

2.6. Funcionalidades Diferenciales de la Interfaz

Se ha diseñado siguiendo los estándares WCAG 2.1 AA. Se utilizan etiquetas semánticas HTML5, atributos ARIA donde es necesario, y un esquema de colores con suficiente contraste. El diseño es totalmente responsive gracias a las clases de utilidad de Tailwind CSS.

3. RESULTADOS

3.1. Pantalla de Listado y Filtros de Eventos

Se ha implementado una pantalla principal donde se muestran los eventos de running obtenidos desde la API. La interfaz permite aplicar filtros por ciudad, fecha, dificultad y otros criterios relevantes, actualizando el listado de forma dinámica.

Ejemplo de estructura de datos consumida por el frontend: { "id": "653a...", "title": "Carrera Popular Madrid", "date": "2025-11-15T09:00:00.000Z", "location": { "city": "Madrid" }, "difficulty": "MEDIUM", "_count": { "registrations": 45 } }

3.2. Flujo de Inscripción y Gestión de Usuario

El frontend guía al usuario a través de un flujo de autenticación (login / registro) y de inscripción a eventos. La aplicación muestra mensajes claros de éxito o error y valida los campos del formulario antes de enviar los datos al backend.

Ejemplo de payload enviado desde el frontend al inscribirse: { "eventId": "653a...", "paceGroup": "5:00 min/km", "emergencyContact": { "name": "María García", "phone": "+34 600 123 456" } }

3.3. Validación, Usabilidad y Pruebas de Interfaz

Se han realizado pruebas manuales completas de la interfaz, verificando los flujos principales: navegación entre pantallas, aplicación de filtros, autenticación, inscripción y mensajes de error. Además, se ha revisado el comportamiento en diferentes tamaños de pantalla para garantizar una experiencia responsiva.

Resumen de Resultados de la Interfaz

  • Navegación: Menú y rutas internas funcionando correctamente.
  • Flujo de eventos: Listado, detalle e inscripción integrados con la API.
  • Experiencia de usuario: Formularios validados y mensajes de feedback claros.

    5. GLOSARIO

    SPA (Single Page Application)
    Aplicación web que carga una sola página HTML y actualiza dinámicamente el contenido a medida que el usuario interactúa con ella, sin recargar la página completa.

    Componente
    Bloque de construcción fundamental en React. Es una pieza de código reutilizable que representa una parte de la interfaz de usuario.

    Hook
    Función especial de React que permite "engancharse" a las características de React (como el estado y el ciclo de vida) desde componentes funcionales.

    Virtual DOM
    Representación ligera en memoria del DOM real. React lo utiliza para optimizar las actualizaciones de la interfaz, minimizando las manipulaciones directas del DOM.

    Context API
    Mecanismo de React para compartir valores (como el usuario autenticado o el tema) entre componentes sin tener que pasar props manualmente en cada nivel.
