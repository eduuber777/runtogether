1. INTRODUCCIÓN

1.1. Contexto y Objetivos Globales

RunTogether nace como una solución integral para la gestión de eventos deportivos, facilitando la conexión entre organizadores y corredores. A lo largo del curso, se ha abordado el desarrollo desde el análisis inicial hasta la implementación técnica completa.

El objetivo final ha sido construir una aplicación web moderna, escalable y segura, capaz de gestionar usuarios, eventos e inscripciones en tiempo real.

1.2. Alcance del Proyecto Final

Esta PEC 4 representa el cierre del ciclo de desarrollo, integrando los módulos de backend y frontend en una aplicación funcional desplegada en un entorno de producción accesible públicamente.

2. ARQUITECTURA FULL STACK

2.1. Diagrama Global

La aplicación sigue una arquitectura cliente-servidor desacoplada. El frontend (SPA) se comunica con el backend a través de peticiones HTTP (API REST), y este a su vez interactúa con la base de datos en la nube.

Flujo de Datos:

Usuario -> Frontend (Netlify) -> API REST (Render) -> Base de Datos (MongoDB Atlas)

2.2. Tecnologías Utilizadas

MongoDB

Base de Datos NoSQL

Express / Node.js

Backend API

React

Frontend UI

Node.js

Runtime Environment

3. DESARROLLO DEL BACKEND (RESUMEN PEC 2)

3.1. API REST y Endpoints

El backend, desarrollado en la PEC 2, expone una API RESTful que gestiona la lógica de negocio. Se han implementado controladores para:

  • Auth: Registro y Login (JWT).
  • Eventos: CRUD completo, búsqueda y filtrado.
  • Inscripciones: Gestión de plazas y registros de usuarios.

3.2. Modelo de Datos

Utilizando Prisma ORM, se definió un esquema robusto que incluye relaciones entre Usuarios, Eventos e Inscripciones, soportando características complejas como grupos de ritmo y detalles técnicos del terreno.

4. DESARROLLO DEL FRONTEND (RESUMEN PEC 3)

4.1. Interfaz de Usuario

La interfaz, construida con React y Tailwind CSS, ofrece una experiencia de usuario fluida y adaptativa (responsive). Se priorizó la usabilidad y la accesibilidad en el diseño de componentes.

4.2. Integración con API

Mediante Axios y un sistema de gestión de estado basado en Context API, el frontend consume los datos del backend de manera eficiente, gestionando la carga, los errores y la autenticación del usuario de forma transparente.

5. INTEGRACIÓN Y DESPLIEGUE (PEC 4)

5.1. Configuración de Entornos

Para asegurar la seguridad y flexibilidad, se utilizaron variables de entorno en ambos extremos:

# Backend (.env) DATABASE_URL="mongodb+srv://..." JWT_SECRET="secreto_seguro" PORT=3000 # Frontend (.env.production) VITE_API_URL="https://runtogether-backend.onrender.com/api"

5.2. Despliegue del Backend en Render

El servicio backend se desplegó en la plataforma Render. Se configuró para desplegarse automáticamente desde el repositorio de GitHub. Se definieron las variables de entorno en el panel de control de Render para la conexión segura a MongoDB Atlas.

5.3. Despliegue del Frontend en Netlify

El cliente React se desplegó en Netlify. El proceso de build (npm run build) genera los archivos estáticos optimizados. Se configuró la redirección de rutas para soportar el routing de la SPA (Single Page Application).

5.4. Pruebas Finales de Integración

Una vez desplegados ambos servicios, se realizaron pruebas E2E (End-to-End) manuales para verificar la correcta comunicación:

  • Registro y Login: Verificado en producción. Token JWT generado y almacenado correctamente.
  • Carga de Eventos: Los datos se recuperan rápidamente desde el servidor en Render.
  • Persistencia: Las inscripciones realizadas desde el frontend desplegado se reflejan inmediatamente en la base de datos.

6. CONCLUSIONES FINALES

El proyecto RunTogether ha culminado con éxito, logrando todos los objetivos académicos y técnicos propuestos. Se ha construido una aplicación Full Stack completa, desde el diseño de la base de datos hasta la interfaz de usuario final.

La integración de tecnologías modernas (MERN Stack) y prácticas de desarrollo profesional (MVC, Componentes, Git Flow) ha resultado en un producto robusto y escalable. El despliegue en la nube demuestra la viabilidad del proyecto en un entorno real.

Como líneas futuras, se podría implementar una pasarela de pagos real (Stripe), notificaciones por email y funcionalidades sociales para compartir eventos.
